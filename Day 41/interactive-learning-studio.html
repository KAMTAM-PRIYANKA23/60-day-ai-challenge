MODULE 01 / 04 — FOUNDATIONS

Classes & Objects

In Java, every object you ever create comes from a class. A class is a template: it describes what data an object will hold (its fields) and what it can do (its methods). The class itself never runs — it just describes the shape. When you write new Car(), Java reads that shape and builds a real, independent object in memory from it.

Analogy A class is a cookie cutter; objects are the cookies. The cutter defines the shape — every cookie will have the same outline — but each cookie is a separate piece of dough. Frost one blue, leave another plain: changing one cookie never changes the cutter, and never changes any other cookie.
The anatomy of a class
public class Car {
    // fields — the data each object holds
    String model;
    int topSpeed;
    int currentSpeed;

    // constructor — runs once, when an object is created
    Car(String model, int topSpeed) {
        this.model = model;
        this.topSpeed = topSpeed;
        this.currentSpeed = 0;
    }

    // method — behavior the object can perform
    void accelerate(int amount) {
        currentSpeed = Math.min(currentSpeed + amount, topSpeed);
    }
}

Three moving parts, always: fields hold state, the constructor sets that state up when an object is born, and methods let the object act or be queried later. The keyword this simply means "the object this code is currently running on" — it's how the constructor tells the difference between the parameter model and the field this.model.

Comparison — class vs. object
ClassObject
The template / blueprint, written once in source codeAn instance built from that template at runtime
Exists only in the compiled program, not in memory as dataLives on the heap, occupies real memory while the program runs
Defines what fields and methods can existHolds actual values in those fields, one set per object

Common misconceptions
Misconception "A class and an object are basically the same thing." They're not — a class is compiled once; you can create thousands of independent objects from it, each with its own field values.
Misconception "Every class needs a constructor I write myself." If you don't write one, Java silently gives you a no-argument default constructor. You only need to write one when you want to set up fields with parameters.
Key takeaways
  • A class defines structure; an object is a live instance of that structure created with new.
  • Fields store per-object state; methods define shared behavior all instances can perform.
  • The constructor runs exactly once, at creation, to set the object's starting state.
  • this refers to the current object and disambiguates fields from parameters of the same name.

MODULE 02 / 04 — CONTROLLED ACCESS

Encapsulation

🔒 Finish Module 1's quiz to unlock

Encapsulation means bundling an object's data together with the code that's allowed to touch it — and then restricting outside access to that data. In Java, you do this with access modifiers: private fields are hidden from the outside world, and public methods (getters, setters, and other behavior) become the only approved doorway in or out.

Analogy Think of an ATM. You never open the machine and move cash around by hand — you use the card slot and keypad. The bank controls what operations are allowed (withdraw, deposit, check balance) and can reject a withdrawal that would overdraw the account. The cash inside is private; the keypad is the public interface.
Without vs. with encapsulation

❌ Public fields — no protection

public class Account {
    public double balance;
}

acct.balance = -500; // legal! nothing stops this

✅ Private field + validated setter

public class Account {
    private double balance;

    public void deposit(double amt) {
        if (amt > 0) balance += amt;
    }
    public double getBalance() {
        return balance;
    }
}

The second version makes an invalid state unrepresentable from the outside. Every change to balance flows through deposit(), so validation lives in exactly one place instead of being trusted to every piece of code that touches an Account.

ModifierVisible from

Common misconceptions
Misconception "Encapsulation just means marking fields private." Private without any public accessor makes a field completely unusable from outside. Encapsulation is private data plus a deliberate, validated public interface — the hiding and the doorway both matter.
Misconception "Getters and setters should exist for every field automatically." Auto-generating a setter for every field just re-exposes the data with extra steps. Only add a setter if outside code legitimately needs to change that value — and validate inside it.
Key takeaways
  • Encapsulation hides internal state (private) and exposes only deliberate, validated operations (public methods).
  • Access modifiers form a spectrum from private to public, with protected and default sitting in between.
  • Validation logic belongs inside setters/methods, not scattered across every place that uses the object.
  • Good encapsulation makes invalid states hard or impossible to create from outside the class.

MODULE 03 / 04 — BUILDING HIERARCHIES

Inheritance

🔒 Finish Module 2's quiz to unlock

Inheritance lets one class acquire the fields and methods of another using extends, expressing an "is-a" relationship. A SportsCar is a Vehicle — so instead of rewriting Vehicle's logic, SportsCar inherits it and adds or overrides only what makes it different.

Analogy Think of biological inheritance. A child inherits traits from a parent automatically — eye color, height tendencies — but can still develop traits the parent never had. In Java, a subclass automatically gets its superclass's fields and methods, and can add new ones or override inherited ones with its own version. 
public class Vehicle {
    protected String model;
    protected int topSpeed;

    Vehicle(String model, int topSpeed) {
        this.model = model;
        this.topSpeed = topSpeed;
    }
    String describe() {
        return model + " tops out at " + topSpeed;
    }
}

public class SportsCar extends Vehicle {
    private boolean hasLaunchControl;

    SportsCar(String model, int topSpeed, boolean lc) {
        super(model, topSpeed); // call the Vehicle constructor first
        hasLaunchControl = lc;
    }

    @Override
    String describe() {
        return super.describe() + " (launch control: " + hasLaunchControl + ")";
    }
}

super(...) calls the parent's constructor — it must be the first line if used. super.describe() calls the parent's version of a method you've overridden, so you can extend behavior instead of fully replacing it. Notice model and topSpeed are protected, not private — that's what makes them reachable from SportsCar while still hidden from unrelated classes.

Overriding vs. overloading — don't mix these up
OverridingOverloading
Subclass redefines a method it inherited, same name + same signatureSame class defines multiple methods with the same name but different parameters
Resolved at runtime, based on the object's real typeResolved at compile time, based on argument types
Requires an inheritance relationshipRequires no inheritance — works within one class
@Override String describe()void move() and void move(int mph)

Common misconceptions
Misconception "A subclass can inherit from more than one class, like a family tree." Java classes support only single inheritance — one direct superclass. Multiple "is-a" style contracts are achieved through interfaces instead (Module 4).
Misconception "Private members are inherited and directly usable in the subclass." Private fields/methods exist in the object in memory, but the subclass's own code cannot reference them by name — only inherited protected/public members are directly accessible.
Key takeaways
  • extends creates an "is-a" relationship; the subclass inherits fields and methods from the superclass.
  • super(...) invokes the parent constructor; super.method() invokes the parent's version of an overridden method.
  • Java allows single inheritance of classes only — one direct superclass per class.
  • protected exposes members to subclasses without making them fully public.

MODULE 04 / 04 — MASTERY

Polymorphism & Abstraction

🔒 Finish Module 3's quiz to unlock

Polymorphism means "many forms": code written against a general type — a superclass or an interface — can operate on many different concrete types, and the correct version of a method runs automatically based on the object's actual type at runtime. This is what lets you write one loop that handles a Circle, a Square, and a Triangle without a chain of if/instanceof checks.

Analogy A USB port is one shared interface. You can plug in a keyboard, a mouse, or a flash drive — the port doesn't need to know which. Each device implements the USB contract in its own way. In Java, an interface plays the same role: it defines what operations exist, and each implementing class decides how. 
interface Shape {
    double area(); // contract only — no body
}

abstract class AbstractShape implements Shape {
    abstract String name(); // subclasses must implement
    String summary() { // shared logic, inherited as-is
        return name() + " area = " + area();
    }
}

class Circle extends AbstractShape {
    double r;
    Circle(double r) { this.r = r; }
    @Override double area() { return Math.PI * r * r; }
    @Override String name() { return "Circle"; }
}

// polymorphism in action:
Shape[] shapes = { new Circle(2), new Square(3) };
for (Shape s : shapes) {
    System.out.println(s.area()); // correct area() runs for each real type
}

Abstract class vs. interface
Abstract classInterface
Can hold fields, constructors, and shared implemented methodsTraditionally only method signatures; may have default methods since Java 8
A class can extend only one abstract classA class can implement many interfaces
Use when subclasses share real code, not just a contractUse when unrelated classes need to promise the same capability
Example: AbstractShape sharing summary()Example: Comparable, Runnable, Shape
Common misconceptions
Misconception "Polymorphism is decided by the variable's declared type." It's decided by the object's actual runtime type. Shape s = new Circle(2); still calls Circle's area(), even though s is declared as Shape.
Misconception "Interfaces can never contain any code." Since Java 8, interfaces can have default and static methods with real bodies — though they still can't hold instance fields the way abstract classes can.
Key takeaways
  • Polymorphism lets one method call resolve to different implementations based on the object's real runtime type.
  • Abstract classes share real implementation across subclasses; interfaces define a pure contract multiple unrelated classes can implement.
  • A class can extend one abstract class but implement many interfaces.
  • Together, encapsulation + inheritance + polymorphism let you write code against abstractions instead of concrete types.

Final practical challenge

Build a small Employee Management hierarchy that uses all four pillars deliberately:

  • An abstract class Employee with private fields (name, baseSalary) exposed through getters — encapsulation.
  • Two subclasses, Manager and Developer, that extend Employee and call super(...) in their constructors — inheritance.
  • An abstract method calculateBonus() that each subclass implements differently, plus a shared describe() method built on top of it — abstraction.
  • A loop over an Employee[] array that calls calculateBonus() polymorphically on every element, regardless of concrete type — polymorphism.
abstract class Employee {
    private String name;
    private double baseSalary;
    Employee(String name, double baseSalary) {
        this.name = name; this.baseSalary = baseSalary;
    }
    String getName() { return name; }
    double getBaseSalary() { return baseSalary; }
    abstract double calculateBonus();
    String describe() {
        return name + ": bonus = " + calculateBonus();
    }
}
class Manager extends Employee {
    Manager(String n, double s) { super(n, s); }
    @Override double calculateBonus() { return getBaseSalary() * 0.20; }
}
class Developer extends Employee {
    Developer(String n, double s) { super(n, s); }
    @Override double calculateBonus() { return getBaseSalary() * 0.12; }
}
// polymorphic loop:
Employee[] team = { new Manager("Rae",90000), new Developer("Kim",80000) };
for (Employee e : team) System.out.println(e.describe());

Summary notes
  • Encapsulation hides state behind a validated public interface.
  • Inheritance lets a subclass reuse and extend a superclass's fields and methods via an "is-a" relationship.
  • Abstraction (abstract classes / interfaces) separates what something does from how it's implemented.
  • Polymorphism lets one call site work correctly across many concrete types, resolved at runtime.
  • Together, these four pillars are what make large Java codebases changeable without rewriting everything that touches them.
